Phase 1: Propose a Stack

Every great specification begins as a hypothesis. This is where you, the architect, define that hypothesis.

What is a "Stack Proposal"?

A proposal is the foundational act of defining a new technology combination within the HttpStack ecosystem. You aren't just listing tools; you are selecting a specific version of an **Element** (e.g., `PostgreSQL 16.1`) for each of the core architectural **Layers** (e.g., Database, Server OS, API Framework).

Whether your stack is a time-tested workhorse like **LAMP** (Linux, Apache, MySQL, PHP) or an unorthodox experiment like **FEVER** (FastAPI, EdgeDB, Vue, Edge Functions, Rust), the proposal process is the same. It stakes a flag in the ground and invites the community to explore its potential.

The Goal of This Phase

To create a clear, unambiguous definition of the stack's components, which becomes the blueprint for the specification that follows. A successful proposal creates a new, empty "spec document" ready to be filled in by the community.

Phase 2: Build the Specification via Q&A

This is the core of HttpStack. We transform unstructured knowledge into a highly-structured, queryable database by asking the right questions.

Beyond the Tutorial

Instead of a single developer writing a monolithic "how-to" guide, the community builds the specification piece by piece. Our system generates hundreds of targeted questions for each stack, covering every layer of interaction:

  • Configuration: "What is the standard `nginx.conf` directive for proxying requests to the Node.js server in this stack?"
  • Interoperability: "Which version of the `pg` library for Node.js is confirmed compatible with `PostgreSQL 16.1` under `Ubuntu 22.04`?"
  • Gotchas &amp; Pitfalls: "Are there known memory leak issues when using this specific caching library with this version of the API framework?"
  • Performance: "What is the typical request-per-second benchmark for the login endpoint when the database layer is under moderate load?"

The Power of Atomic Knowledge

Each answer is a discrete, "atomic" piece of knowledge tagged to the specific elements and layers it concerns. This allows knowledge to be aggregated. An answer about configuring Nginx for a Node.js app contributes not only to the current stack's specification but also to the global knowledge base for Nginx and Node.js.

Phase 3: Verification and Validation

A specification is only a theory until it's proven in practice. Verification is the process of turning theory into fact.

What is a "Verification"?

A verification is a formal report submitted by a community member who has attempted to build and deploy the stack according to its specification. It's the ultimate form of peer review.

To submit a verification, a user must provide evidence, which can include:

  • Deployment Logs: Evidence of a successful build and run process.
  • Repository Links: A link to a public Git repository with a working implementation.
  • Performance Benchmarks: Screenshots or logs from tools like `ab` (Apache Benchmark) or `k6` that validate performance claims.
  • Error Reports: Detailed reports of where the specification was inaccurate or incomplete, helping to refine it.

A verified specification is one that has been independently reproduced by other developers. It's a stamp of approval that says, "This works as described."

The Visual Language: Color Bands of Completeness

We translate the complex state of a specification into a simple, at-a-glance visual: the Stack Band.

Each stack is represented by a horizontal band of colors, where each color corresponds to a specific architectural layer (e.g., Database, Server, API). The **opacity or "richness"** of each color tells you the story of its documentation:

  • Faint/Transparent: The specification for this layer is just beginning. Only a few basic questions have been answered.
  • Semi-Opaque: The spec is well underway. Key configuration and setup questions are answered.
  • Solid/Vibrant: The specification for this layer is comprehensive, covering edge cases, performance tuning, and security.
  • Glowing/Highlighted Border: This layer has been successfully **verified** by at least one other developer. It's battle-tested.

This system allows you to instantly assess the maturity and reliability of any stack in our ecosystem before you invest a single second in trying to build it.
